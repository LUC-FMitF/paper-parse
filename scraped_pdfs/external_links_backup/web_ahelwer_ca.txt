      <article role="main" class="blog-post">
        <p>Last weekend I had a conversation with an undergraduate student new to computer science, who was reading <a href="https://en.wikipedia.org/wiki/Introduction_to_Algorithms">CLRS</a>.
&ldquo;I wish&rdquo; they said, &ldquo;that all the pseudocode in my algorithms textbook was just written in Python.&rdquo;
&ldquo;Ah&rdquo; I said, &ldquo;but textbook authors sometimes want their work to endure beyond a decade.&rdquo;
&ldquo;But Python&rsquo;s been around for a long time&rdquo; came the reply, &ldquo;and it&rsquo;s very readable, and you can&rsquo;t execute pseudocode anyway so what&rsquo;s the harm?&rdquo;</p>
<p>It&rsquo;s true that Python has been around for three decades and popular for at least two of them, although post-2000 non-pseudocode-based textbooks generally went with Java instead.
I do feel that tying algorithms - these immortal mathematical objects - to languages relevant within a half-century sliding window cheapens their presentation somehow.
I&rsquo;m old enough to have lived through the waning of several once-popular languages.
Pseudocode is timeless!
Its lack of standardization &amp; executability is arguably a problem; this presumably is why Donald Knuth defined <a href="https://en.wikipedia.org/wiki/MMIX">his own assembly language</a> running on an imaginary computer architecture to use in <em>The Art of Computer Programming</em>.</p>
<p>But what if pseudocode could itself be made executable?
Not only that, what if formal proofs could be written about its correctness that connect directly to the definitions in the pseudocode itself?
This was the basic idea behind PlusCal, a pseudocode-like language developed by Leslie Lamport that transpiles to TLA⁺.
PlusCal sees most of its use in distributed &amp; concurrent systems, but I wanted to see how it handled pseudocode of the sort found in CLRS or published papers.
I wanted to know:</p>
<ol>
<li>Do algorithms written in PlusCal look like their pseudocode counterparts?</li>
<li>Can we execute the algorithms written in PlusCal without many alterations from the form most resembling the original pseudocode? Can we easily find the bugs?</li>
<li>How difficult is it to write formal proofs of correctness for PlusCal pseudocode?</li>
<li>How readable are those proofs compared to the informal proof given in the paper?</li>
</ol>
<p>I also wanted to see how Python performed on the first two questions.
Let&rsquo;s find out!</p>
<h1 id="the-algorithm">The Algorithm</h1>
<p>For our case study we&rsquo;re going to stretch back in time to a paper published in 1980, now 43 years old - 10 years older than the first edition of CLRS.
The paper is <a href="https://doi.org/10.1016/0020-0190(80)90149-0"><em>Lexicographically Least Circular Substrings</em></a> by <a href="https://www.cs.ubc.ca/~ksbooth/">Kellogg S. Booth</a>, now-retired Professor Emeritus at the Department of Computer Science at the University of British Columbia.
I knew this paper from my time in competitive programming during undergrad.
Specifically I knew the pseudocode given in the paper was wrong; the corrected version of the code was disseminated within the competitive programming community.
There&rsquo;s an interesting aside here about technical correctness vs. general correctness in published results; the algorithm in the paper is literally wrong, even though it comes with a proof of correctness, but the paper is still a good paper because the <em>fundamental insight</em> driving the algorithm is correct, and the technical problems can be fixed.
So if you hear a statistic like &ldquo;X% of math/computer science papers contain significant technical errors&rdquo; that is almost certainly true, but those errors aren&rsquo;t necessarily fatal.
These are the sort of errors that are recorded via publication of errata, not retraction.
Actually there is <a href="https://www.cs.ubc.ca/~ksbooth/PUB/LCS.shtml">an errata page</a> for this very paper, but the amended algorithm is still not correct!</p>
<p>So what does the algorithm actually do?
Given a string, the algorithm outputs the rotation of the string that comes first alphabetically - the lexicographically-least circular substring.
For example, the string <code>&quot;abca&quot;</code> has the rotation <code>&quot;aabc&quot;</code> which is first alphabetically among its four possible rotations:</p>
<div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;"><code class="language-bash" data-lang="bash"><span style="display:flex;"><span><span style="color:#e6db74">&#34;abca&#34;</span>
</span></span><span style="display:flex;"><span><span style="color:#e6db74">&#34;bcaa&#34;</span>
</span></span><span style="display:flex;"><span><span style="color:#e6db74">&#34;caab&#34;</span>
</span></span><span style="display:flex;"><span><span style="color:#e6db74">&#34;aabc&#34;</span>
</span></span></code></pre></div><p>The algorithm outputs a positive number denoting the number of times the input string must be rotated to the left to reach its lexicographically-minimal rotation.
For <code>&quot;abca&quot;</code>, this is 3.
Some strings, like <code>&quot;baba&quot;</code>, have multiple lexicographically-minimal rotations (1 and 3); in these cases, the algorithm outputs the smallest rotation (1).
The algorithm outputs 0 when given strings of length zero or one.
Given a string of length \(n\), the algorithm has \(O(n)\) time &amp; memory complexity.
The naive algorithm of generating every possible string rotation and comparing them has \(O(n^2)\) time complexity.</p>
<p>Where is this algorithm used?
Beyond contrived programming competition problems, perhaps you have a database of <a href="https://en.wikipedia.org/wiki/Circular_DNA">circular DNA</a> strings and you want to store them in normalized form for easier searching.
The problem also crops up when normalizing cycles generally, for example cycles of a graph. It is also found in various places <a href="https://en.wikipedia.org/wiki/Lyndon_word">in mathematics</a>.</p>
<p>What is the actual algorithm?
Just to set expectations, it is short but very difficult to understand.
It is a modification of the famously-inscrutable pre-process step of the <a href="https://en.wikipedia.org/wiki/Knuth%E2%80%93Morris%E2%80%93Pratt_algorithm">Knuth-Morris-Pratt string search algorithm</a>.
Thankfully for the purpose of this post you don&rsquo;t need to understand why the algorithm actually works.
Here is the paper&rsquo;s pseudocode transcribed verbatim; the input string&rsquo;s characters are denoted by \(b_k\), while \(n\) denotes the string&rsquo;s length.
\(f\) is an array in which we store some scratch-work.
Behold a labyrinth of index arithmetic!</p>
<p>$$
\begin{aligned}
&amp;\text{integer procedure } LCS(b_0 \ldots b_{n-1})\text{:} \cr
&amp;\quad\text{begin} \cr
1.&amp;\quad\quad f(0) \gets nil; \cr
2.&amp;\quad\quad k \gets 0; \cr
3.&amp;\quad\quad \text{for } j \gets 1 \text{ to } 2n-1 \text{ do} \cr
&amp;\quad\quad\quad \text{begin} \cr
4.&amp;\quad\quad\quad\quad \text{if } j - k \geq n \text{ then return } k; \cr
5.&amp;\quad\quad\quad\quad i \gets f(j - k - 1); \cr
6.&amp;\quad\quad\quad\quad \text{while } b_{j \pmod{n}} \neq b_{k+i+1 \pmod{n}} \text{ and } i \neq nil \text{ do} \cr
&amp;\quad\quad\quad\quad\quad \text{begin} \cr
7.&amp;\quad\quad\quad\quad\quad\quad \text{if } b_{j \pmod{n}} &lt; b_{k+i+1 \pmod{n}} \text{ then} \cr
8.&amp;\quad\quad\quad\quad\quad\quad\quad k \gets j - i - 1; \cr
9.&amp;\quad\quad\quad\quad\quad\quad i \gets f(i); \cr
&amp;\quad\quad\quad\quad\quad \text{end} \cr
10.&amp;\quad\quad\quad\quad \text{if } b_{j \pmod{n}} \neq b_{k+i+1 \pmod{n}} \text{ and } i = nil \text{ then} \cr
&amp;\quad\quad\quad\quad\quad \text{begin} \cr
11.&amp;\quad\quad\quad\quad\quad\quad \text{if } b_{j \pmod{n}} &lt; b_{k+i+1 \pmod{n}} \text{ then} \cr
12.&amp;\quad\quad\quad\quad\quad\quad\quad k \gets j; \cr
13.&amp;\quad\quad\quad\quad\quad\quad f(j - k) \gets nil; \cr
&amp;\quad\quad\quad\quad\quad \text{end} \cr
&amp;\quad\quad\quad\quad \text{else} \cr
14.&amp;\quad\quad\quad\quad\quad f(j - k) \gets i + 1; \cr
&amp;\quad\quad\quad \text{end} \cr
&amp;\quad \text{end} \cr
\end{aligned}
$$</p>
<p>The algorithm has a nested loop.
Naively you&rsquo;d expect this to mean it has \(O(n^2)\) time complexity instead of \(O(n)\).
Not so!</p>
<h1 id="translation">Translation</h1>
<p>Reading the pseudocode, a few things pop out:</p>
<ol>
<li>The pseudocode is inconsistent in using begin/end statements versus indentation to encapsulate code blocks; in the original paper I believe this is due to column width constraints, but one has to be careful to write the instructions in the correct block.</li>
<li>There is only one return statement: the one in instruction 4.
If the loop actually terminates the return value is undefined; perhaps the loop is guaranteed to return in instruction 4 before termination but most programming languages can&rsquo;t be convinced of this and would refuse to compile it.</li>
<li>Mixing in the value \(nil\) with integer values in function \(f\); will this cause problems?</li>
<li>Initialization: how many values does \(f\) need to store?
What are the values of \(f\) initialized to?
What is the initial value of variable \(i\)?</li>
</ol>
<p>These are all very typical issues when translating pseudocode.
We handle them as follows:</p>
<ol>
<li>Check your code, then check it again. No really! I&rsquo;ve yet to translate this algorithm without making at least one mistake.</li>
<li>This is left out of the pseudocode for some reason, but reading the rest of the paper it&rsquo;s made clear that \(k\) is the final result value we&rsquo;re looking for.
So the end of the pseudocode really should look like:
$$
\begin{aligned}
&amp;\quad\quad\quad \text{end} \cr
15.&amp;\quad\quad \text{return } k; \cr
&amp;\quad \text{end} \cr
\end{aligned}
$$</li>
<li>The value \(nil\) does indeed cause problems.
If you trace execution for a simple string like <code>&quot;ab&quot;</code>, you&rsquo;ll see that in instruction 6 the value \(b_{k+i+1 \pmod{n}}\) is calculated when \(i\) has value \(nil\).
Even short-circuiting that calculation by checking \(i \neq nil\) first only shifts the problem to instruction 10.
The solution, somewhat inscrutably, is to make \(nil\) have the value \(-1\).</li>
<li>Some experimentation shows \(f\) needs to be initialized with \(2n\) values, all set to \(-1\).
Variable \(i\) is contained within the scope of the for loop, and is initialized in instruction 5.</li>
</ol>
<p>So quite a bit of elbow grease is necessary to understand how a translated program would look.</p>
<h3 id="python">Python</h3>
<p>Here&rsquo;s a straightforward Python translation:</p>

</span></span></code></pre></div><p>It actually looks pretty similar to the pseudocode!
Very readable.
The code seems to give correct results without any crashes on a few test strings.</p>
<h3 id="pluscal">PlusCal</h3>
<p>If you get into string algorithms, you quickly learn to dread off-by-one errors.
This is compounded by about half the original sources using 0-indexed strings, and the other half using 1-indexed strings;
This is <em>further</em> compounded because most programming languages have 0-indexed strings &amp; arrays, except for a few like Fortran and&hellip; PlusCal &amp; TLA⁺.
Fortunately the immense flexibility of PlusCal &amp; TLA⁺ makes defining our own 0-indexed string type simple, to match the 0-indexed presentation of our chosen algorithm.
I&rsquo;ll omit that code below for simplicity.
Here&rsquo;s the PlusCal translation I came up with:</p>
<div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;"><code class="language-julia" data-lang="julia"><span style="display:flex;"><span><span style="color:#f92672">--</span>algorithm LeastCircularSubstring

</span></span><span style="display:flex;"><span>      <span style="color:#66d9ef">else</span>

</span></span><span style="display:flex;"><span>LVR<span style="color:#f92672">:</span>  j <span style="color:#f92672">:=</span> j <span style="color:#f92672">+</span> <span style="color:#ae81ff">1</span>;

It looks even better typeset; TLA⁺ has a command to translate specs to \(\LaTeX\) output.
You can see the result of that as a PDF <a href="https://github.com/tlaplus/Examples/blob/6d47e1280712b66f7f0c94e772a084fe0c058b37/specifications/LeastCircularSubstring/LeastCircularSubstring.pdf">here</a>.
I think it would have been neat to typeset the <code>% n</code> operator as \(\pmod{n}\), but it still looks nice.</p>
<p>A few things to note about the PlusCal code:</p>
<ul>
<li>The operators <code>ZLen(s)</code> and <code>ZSeq(S)</code> are defined in a <code>ZSequences</code> module I wrote for zero-indexed sequences; <code>ZSeq(S)</code> is the set of all strings over the character set <code>S</code>. The line <code>b \in Corpus</code> nondeterministically initializes <code>b</code> as an arbitrary element of this set. <code>CharacterSet</code> is an input to this spec.</li>
<li>In PlusCal you can label statements to refer to the statements in later proofs or invariants; I have set these to correspond to the pseudocode instruction numbers. The <code>Done</code> label is a built-in label denoting the program state upon termination.</li>
<li>PlusCal doesn&rsquo;t have a notion of variable scope; all variables are global. It also doesn&rsquo;t have for-loops, only while-loops. We have an extra instruction, labeled <code>LVR</code> for loop variant, that increments <code>j</code> at the bottom of the loop.</li>
<li>This algorithm is written without return statements; similar to the pseudocode, we just look at the value of <code>k</code> after the algorithm has terminated.</li>
</ul>
<p>There&rsquo;s one more weird thing about PlusCal.
I mentioned it transpiles to TLA⁺; it actually does this <em>inside the source file itself</em>.
A block is set aside below this code for the corresponding TLA⁺ code to be emitted.
I used to really dislike this.
Not only does it involve generated code, but generated code inside a human-written source file!?
This turns out to make sense when you&rsquo;re writing proofs and invariants over this code in TLA⁺ itself.
I&rsquo;d be interested in alternatives, though.</p>
<h1 id="execution--testing">Execution &amp; Testing</h1>
<h3 id="python-1">Python</h3>
<p>The Python code executes as-is, so let&rsquo;s find those bugs!
We could dream up a bunch of tricky strings, but we live in the modern era so let&rsquo;s act like modern people and use property-based testing.
We will generate a ton of random strings, get the output of our algorithm, and compare it to the naive &ldquo;check every rotation&rdquo; approach.
Here&rsquo;s that naive approach in Python:</p>
<div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;"><code class="language-py" data-lang="py"><span style="display:flex;"><span><span style="color:#66d9ef">def</span> <span style="color:#a6e22e">naive_lcs</span>(s):
</span></span><span style="display:flex;"><span>    n <span style="color:#f92672">=</span> len(s)
</span></span><span style="display:flex;"><span>    <span style="color:#66d9ef">if</span> n <span style="color:#f92672">==</span> <span style="color:#ae81ff">0</span>:
</span></span><span style="display:flex;"><span>        <span style="color:#66d9ef">return</span> <span style="color:#ae81ff">0</span>
</span></span><span style="display:flex;"><span>    rotations <span style="color:#f92672">=</span> [s[i:] <span style="color:#f92672">+</span> s[:i] <span style="color:#66d9ef">for</span> i <span style="color:#f92672">in</span> range(n)]
</span></span><span style="display:flex;"><span>    least_rotation <span style="color:#f92672">=</span> min(rotations)
</span></span><span style="display:flex;"><span>    <span style="color:#66d9ef">return</span> rotations<span style="color:#f92672">.</span>index(least_rotation)
</span></span></code></pre></div><p>We can then use <a href="https://hypothesis.readthedocs.io/en/latest/index.html">Hypothesis</a> to auto-generate strings from a very simple alphabet, just <code>&quot;0&quot;</code> and <code>&quot;1&quot;</code>:</p>
<div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;"><code class="language-py" data-lang="py"><span style="display:flex;"><span><span style="color:#f92672">from</span> hypothesis <span style="color:#f92672">import</span> given
</span></span><span style="display:flex;"><span><span style="color:#f92672">from</span> hypothesis.strategies <span style="color:#f92672">import</span> text, sampled_from
</span></span><span style="display:flex;"><span>
</span></span><span style="display:flex;"><span><span style="color:#a6e22e">@given</span>(text(sampled_from(<span style="color:#e6db74">&#34;01&#34;</span>)))
</span></span><span style="display:flex;"><span><span style="color:#66d9ef">def</span> <span style="color:#a6e22e">test_lcs</span>(b):
</span></span><span style="display:flex;"><span>    <span style="color:#66d9ef">assert</span> lcs(b) <span style="color:#f92672">==</span> naive_lcs(b)
</span></span></code></pre></div><p>Running <code>pytest</code>, we very quickly find a bug!
The algorithm incorrectly handles the string <code>&quot;010&quot;</code>, returning <code>0</code> when it should return <code>2</code>.
Here we can defer to the <a href="https://www.cs.ubc.ca/~ksbooth/PUB/LCS.shtml">published errata</a>, which says to modify instruction 4, the early return statement, from:
$$
\begin{aligned}
4.&amp;\quad\quad\quad\quad \text{if } j - k \geq n \text{ then return } k; \cr
\end{aligned}
$$
to:
$$
\begin{aligned}
4.&amp;\quad\quad\quad\quad \text{if } j - k &gt; n \text{ then return } k; \cr
\end{aligned}
$$
So using \(&gt;\) instead of \(\geq\).
Let&rsquo;s test it out!
Unfortunately after this fix Hypothesis immediately finds another failing string: <code>&quot;0010&quot;</code>, where we get <code>0</code> when we should get <code>3</code>.
The errata says that the early return in instruction 4 is only a &ldquo;shortcut&rdquo; and isn&rsquo;t necessary for the correctness of the algorithm - let&rsquo;s try removing it to see if that helps:</p>

</span></span><span style="display:flex;"><span>            i <span style="color:#f92672">=</span> f[i]

</span></span><span style="display:flex;"><span>            <span style="color:#66d9ef">if</span> b[j <span style="color:#f92672">%</span> n] <span style="color:#f92672">&lt;</span> b[(k <span style="color:#f92672">+</span> i <span style="color:#f92672">+</span> <span style="color:#ae81ff">1</span>) <span style="color:#f92672">%</span> n]:
</span></span><span style="display:flex;"><span>                k <span style="color:#f92672">=</span> j

<h3 id="pluscal-1">PlusCal</h3>
<p>Python had its fun, now let&rsquo;s see whether PlusCal finds those bugs.
We can use the TLC model checker to exhaustively test every string in a certain character set below a certain length.
First, like Python, we have to define the naive algorithm for lexicographically-minimal string rotation to have something to test against.
We&rsquo;re writing this in TLA⁺ itself, not PlusCal.
Here TLA⁺ really suffers from its minimal standard library; I have to go all the way down to defining what lexicographic order means for my <code>ZSequences</code> module:</p>
<div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;"><code class="language-idris" data-lang="idris"><span style="display:flex;"><span>a <span style="color:#f92672">\</span>preceq b <span style="color:#f92672">==</span>

<h1 id="formal-proof">Formal Proof</h1>
<p>What if we&rsquo;re impatient?
What if we want to know that this algorithm is truly correct for all strings of finite length across all possible character sets?
This algorithm has such a history, with decades of corrections and incorrections and whispered accounts of how it&rsquo;s really done that we deserve to <em>know</em>, once and for all, that this is really the algorithm we&rsquo;re looking for.
No more revising errata, the buck stops here in this very post!
Can we do it?</p>
<p>Okay we probably could do it, but I&rsquo;d estimate it would take at least two weeks.
I actually accomplished a near-decade-long goal and finally taught myself the TLA⁺ proof language for this blog post, hoping to prove the correctness of our algorithm!
For warmup I <a href="https://github.com/ahelwer/Examples/blob/a398fe30dd586a10f738ec04a72c11a73ef8b6e3/specifications/LearnProofs/FindHighest.tla">proved correct</a> a very, very simple algorithm that just iterates through a sequence of numbers to find its greatest element.
That took me three entire days!
The proof was subsequently <a href="https://github.com/ahelwer/Examples/commit/18f0d74b1c89779030877f71a638891236775222#diff-92fb665fe432fc35aa1ea066184b479f17d983edc362f9e0ef0faf1b6139da95">greatly simplified</a> with help from Stephen Merz and future proofs will surely be faster with all the experience I acquired, but still!
This post is about a very complicated algorithm.
So I am sorry to say the buck does not actually stop here.
The buck is passed along to whomever is willing to invest two weeks of their life formally proving this algorithm correct.
That person could still be future me, who knows!
But it is not imminent-future me.</p>
<p>Still, with my experience I can answer the questions about proofs I had at the beginning of the post.
I wanted to know how difficult it was to write formal proofs of correctness for PlusCal pseudocode, and how readable those proofs were.
Difficulty?
Actually quite low.
Proofs about algorithms can be thought of as an inch deep and a mile wide.
You rarely encounter anything more mathematically-challenging than \(a \leq b \land b \leq c \implies a \leq c\).
However you have to prove <em>so many things</em> that the time investment is quite large.
Usually you prove correctness of a program by defining something called an inductive invariant, then demonstrating that every possible code path maintains the correctness of that invariant (the idea being upon termination this invariant proves the correctness of your algorithm).
So I&rsquo;d be writing a fairly detailed proof for every instruction in our program, of which there are twelve, and some of those have branches which would themselves require sub-proofs.
A lot of work!
Once you learn the quirks of the TLA⁺ proof manager it&rsquo;s definitely tractable, though.</p>
<p>How about proof readability?
I would say quite readable!
Mileage may vary but the individual proof steps are simple, and once you understand what&rsquo;s being proved &amp; how proofs are usually structured it&rsquo;s easy to pick out the interesting parts.
For example, with my &ldquo;find the greatest element in a sequence&rdquo; algorithm it&rsquo;s clear the most interesting part of the proof happens when we compare the current value of the sequence with the current highest recorded value, and choose one or the other.
That part of the proof is <a href="https://github.com/ahelwer/Examples/blob/a398fe30dd586a10f738ec04a72c11a73ef8b6e3/specifications/LearnProofs/FindHighest.tla#L134:L148">here</a>.
The rest of the proof is boilerplate by comparison: necessary for the proof manager to give us its stamp of correctness, but intellectually not too interesting.
Still, even if you don&rsquo;t know TLA⁺ that proof is fairly readable!
You can see we&rsquo;re showing that the index element <code>f[i] &lt;= h'</code>, the highest recorded value, where <code>h' = max(h, f[i])</code> (the <code>'</code> means &ldquo;prime&rdquo; and denotes the value of the variable in the next state).</p>
<p>TLA⁺ proofs are very interesting when compared to another proof language I&rsquo;ve used, Lean.
The experience of writing Lean proofs is <em>wonderful</em>.
There is never any confusion about what the current proof goal is, what assumptions you can use, and what moves you can make.
However, Lean proofs as written are absolutely unintelligible.
Like what the hell does this mean!?</p>

You&rsquo;ve heard about programming paradigms, now you&rsquo;re gonna learn about proof paradigms!
Lean proofs only make sense if you&rsquo;re reading them inside <a href="https://leanprover-community.github.io/lean-web-editor/#code=import%20tactic%0A%0Adef%20sum_of_first_n_nat%20%3A%20%E2%84%95%20%E2%86%92%20%E2%84%95%0A%7C%200%20%3A%3D%200%0A%7C%20%28nat.succ%20n%29%20%3A%3D%20%28nat.succ%20n%29%20%2B%20sum_of_first_n_nat%20n%0A%0A%23eval%20sum_of_first_n_nat%204%0A%0Atheorem%20closed_eq_sum_of_first_n_nat%20%28n%20%3A%20%E2%84%95%29%20%3A%0A%20%20%20%202%20*%20%28sum_of_first_n_nat%20n%29%20%3D%20n%20*%20%28nat.succ%20n%29%20%3A%3D%0Abegin%0Ainduction%20n%20with%20d%20hd%2C%0A%20%20rw%20sum_of_first_n_nat%2C%0A%20%20rw%20nat.mul_zero%2C%0A%20%20rw%20nat.zero_mul%2C%0Arw%20sum_of_first_n_nat%2C%0Arw%20nat.left_distrib%2C%0Arw%20hd%2C%0A--rewrites%20nat.succ%20n%20to%20n%20%2B%201%3A%0Arw%20nat.succ_eq_add_one%2C%0A--rewrites%20nat.succ%20%28n%20%2B%20m%29%20to%20n%20%2B%20nat.succ%20m%20%28from%20defn%20of%20addition%29%0A--note%20rw%20usually%20rewrites%20from%20left%20to%20right%20over%20an%20equality%3B%20%E2%86%90%20%28%5Cl%29%20does%20right%20to%20left%0Arw%20%E2%86%90%20nat.add_succ%2C%0A--rewrites%20nat.succ%201%20to%202%0Arw%20%28show%20nat.succ%201%20%3D%202%2C%20by%20refl%29%2C%0A--multiplies%20out%20d%20%2B%201%0Arw%20left_distrib%20%28d%20%2B%201%29%20d%202%2C%0A--moving%20things%20around%20with%20commutativity%0Arw%20mul_comm%202%20%28d%20%2B%201%29%2C%0Arw%20mul_comm%20d%20%28d%20%2B%201%29%2C%0Arw%20add_comm%2C%0Aend%0A%0Atheorem%20closed_eq_sum_of_first_n_nat_with_ring%20%28n%20%3A%20%E2%84%95%29%20%3A%0A%20%20%20%202%20*%20%28sum_of_first_n_nat%20n%29%20%3D%20n%20*%20%28nat.succ%20n%29%20%3A%3D%0Abegin%0Ainduction%20n%20with%20d%20hd%2C%0A%20%20rw%20sum_of_first_n_nat%2C%0A%20%20ring%2C%0Arw%20sum_of_first_n_nat%2C%0Arw%20nat.left_distrib%2C%0Arw%20hd%2C%0Aring%2C%0Aend">a Lean programming environment</a>.
As you move the cursor through the text, a window on the right shows how each instruction transforms the proof goals or assumptions.
By contrast, TLA⁺ proofs make sense in and of themselves.
The tradeoff here is they are much harder to write!
It&rsquo;s never quite clear why the backend provers are having trouble proving something, and the only way to figure it out is to break a statement down into further proofs and subproofs until you realize it just didn&rsquo;t know it could assume some variable was an integer, so you delete like 15 lines and replace them with a single statement.
However, the end result is a structured proof that is readable as written, without accompanying software.</p>
<h1 id="conclusion">Conclusion</h1>
<p>I&rsquo;ll admit, Python made a very compelling case for itself here.
The final algorithm, naive algorithm, and property-based testing code combined took up only 35 very readable lines!
I&rsquo;ve been convinced, immortal mathematical objects be damned, Python is a perfectly fine alternative to pseudocode in papers and textbooks.
Of course I would suggest sticking to a very restricted subset of Python, probably avoiding any fancy features (no list comprehensions, pattern matching, etc.) and not using the standard library.
I wonder whether such a restrictive subset of Python has been standardized.
Perhaps it is simply Python 1.0!</p>
<p>The verdict for PlusCal is a bit more ambiguous.
It gained a lot of verbosity from having 1-indexed arrays, an extremely minimal standard library, and in-file code generation.
It ran around 275 lines (100 of those generated) split across four source files just to match Python&rsquo;s 35-line execution &amp; test functionality.
Also, it took more effort to write: probably an entire day of work for the basics plus another day for polish, vs. less than an hour for the Python version.</p>
<p>To be fair, sequential string algorithms are not PlusCal&rsquo;s strength.
It really shines in concurrent &amp; distributed algorithms, where the model checker explores all possible interleavings of instructions.
PlusCal also has two killer features here for paper &amp; textbook authors:</p>
<ol>
<li>It can output paper-ready \(\LaTeX\) pseudocode directly from the code itself, so the code that&rsquo;s published is the same as the code that&rsquo;s been parsed &amp; model-checked</li>
<li>The formal proof system directly checks that same code</li>
</ol>
<p>I don&rsquo;t think it would be too hard to write a \(\LaTeX\) generator for a restricted subset of Python code (if such a thing would even be desirable), but Python doesn&rsquo;t have model checking.
Hypothesis is wonderful for sequential algorithms, but nothing can really compare to TLA⁺ model checking for quickly testing designs of concurrent &amp; distributed systems.</p>
<p>What about proofs?
Here&rsquo;s a probably-not-correct hot take: proofs will be the future of TLA⁺ relevance.
Model checking was the killer app for the past decade, but property-based testing, model-based testing, and fuzzing have come <em>so far</em> in that time and they operate on the actual running system instead of its abstract specification.
MongoDB worked with an apparently impressive stealth-mode company in this space called Antithesis that set up <a href="https://github.com/mongodb/mongo/wiki/Testing-MongoDB-with-Antithesis">very effective network fuzzing</a> for their database.
Everybody&rsquo;s also heard of <a href="https://jepsen.io/analyses">Jepsen</a>.
However, TLA⁺ can retreat to infinity and prove an algorithm correct for all possible inputs!
It also works well in tandem with the model checker, as you can quickly check the validity of a statement before proving it (to quote Leslie Lamport, it&rsquo;s easier to prove something if it&rsquo;s true!)
Given some UX improvement I could see proofs becoming a viable option, especially if everybody gets <a href="https://groups.google.com/g/tlaplus/c/3WBSL-IZyzU/m/1SQ8fdZGCAAJ">a LLM tutor</a> to walk them through the learning curve.</p>
<p>Okay, so what&rsquo;s the final word?
Should you use PlusCal for pseudocode in your paper?
Yes, that or Python.
I have come to believe that both are superior to an ad-hoc unspecified pseudo-math language.
If you want to see PlusCal pseudocode in an actually published paper, take a look at <a href="https://arxiv.org/abs/1703.08905">the WPaxos paper</a> by Ailidani Ailijiang, Aleksey Charapko, Murat Demirbas, and Tevfik Kosar.</p>
